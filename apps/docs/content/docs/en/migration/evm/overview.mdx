# A Complete Guide to Solana Development for Ethereum Developers

Get started with the key differences between EVM and SVM, and build your first
Solana program.

## The Mental Model Shift

The single most important concept to internalize: **Solana programs are
stateless**. All state lives in separate accounts owned by the program.

| Ethereum                      | Solana                                 |
| ----------------------------- | -------------------------------------- |
| Contract stores its own state | Program reads/writes external accounts |
| `storage` keyword             | Account data field                     |
| `mapping(address => uint)`    | PDA per user                           |
| `msg.sender`                  | `ctx.accounts.signer.key()`            |
| `address(this)`               | `ctx.program_id`                       |

## Your First Program

### Counter: Solidity vs. Anchor

**Solidity:**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract Counter {
    mapping(address => uint256) public counts;

    function increment() public {
        counts[msg.sender] += 1;
    }
}
```

**Anchor (Rust):**

```rust
use anchor_lang::prelude::*;

declare_id!("YourProgramId...");

#[program]
pub mod counter {
    use super::*;

    pub fn initialize(ctx: Context<Initialize>) -> Result<()> {
        ctx.accounts.counter.count = 0;
        ctx.accounts.counter.owner = ctx.accounts.user.key();
        Ok(())
    }

    pub fn increment(ctx: Context<Increment>) -> Result<()> {
        ctx.accounts.counter.count += 1;
        Ok(())
    }
}

#[derive(Accounts)]
pub struct Initialize<'info> {
    #[account(
        init,
        payer = user,
        space = 8 + 8 + 32,
        seeds = [b"counter", user.key().as_ref()],
        bump
    )]
    pub counter: Account<'info, Counter>,
    #[account(mut)]
    pub user: Signer<'info>,
    pub system_program: Program<'info, System>,
}

#[derive(Accounts)]
pub struct Increment<'info> {
    #[account(
        mut,
        seeds = [b"counter", user.key().as_ref()],
        bump,
        has_one = owner
    )]
    pub counter: Account<'info, Counter>,
    pub user: Signer<'info>,
}

#[account]
pub struct Counter {
    pub count: u64,
    pub owner: Pubkey,
}
```

### Key differences explained

1. **`seeds` + `bump`** — This is how you get a PDA. Instead of
   `mapping(address => Counter)`, each user has their own Counter account at a
   deterministic address.

2. **`space = 8 + 8 + 32`** — Solana accounts must pre-allocate space. The `8`
   is Anchor's account discriminator, `8` is `u64`, `32` is `Pubkey`.

3. **`has_one = owner`** — Anchor constraint ensuring only the original
   initializer can increment.

## Calling Your Program from TypeScript

```typescript
import * as anchor from "@coral-xyz/anchor";

const provider = anchor.AnchorProvider.env();
anchor.setProvider(provider);
const program = anchor.workspace.Counter as anchor.Program<Counter>;

const [counterPda] = anchor.web3.PublicKey.findProgramAddressSync(
  [Buffer.from("counter"), provider.wallet.publicKey.toBuffer()],
  program.programId
);

await program.methods
  .initialize()
  .accounts({ counter: counterPda, user: provider.wallet.publicKey })
  .rpc();

await program.methods
  .increment()
  .accounts({ counter: counterPda, user: provider.wallet.publicKey })
  .rpc();

const counterAccount = await program.account.counter.fetch(counterPda);
console.log("Count:", counterAccount.count.toString());
```

## Key Concepts Summary

| Concept          | Ethereum                 | Solana                       |
| ---------------- | ------------------------ | ---------------------------- |
| State storage    | Inside contract          | External accounts (PDAs)     |
| Token standard   | ERC20 (custom contract)  | SPL Token (shared program)   |
| NFT standard     | ERC721 (custom contract) | Metaplex                     |
| Events           | `emit Event()`           | Account state / logs         |
| Reentrancy guard | Manual `nonReentrant`    | Not needed (stateless)       |
| Gas              | ETH-based                | Compute units (SOL fees)     |
| Upgradability    | Proxy pattern            | Program upgrades (authority) |

## FAQ

### What is the biggest architectural difference between EVM and SVM?

On Ethereum, smart contracts store their own state. On Solana, programs are
stateless — all state lives in separate accounts owned by the program. This
separation enables Solana's parallel transaction execution and is the core
concept to internalize before migrating.

### Can I deploy my Solidity contracts on Solana?

Not directly. Solana programs are written in Rust (or other languages compiled
to BPF bytecode). However, projects like [Solang](https://www.solang.dev/) can
compile some Solidity to BPF. For most projects, a full rewrite in Rust is the
recommended approach.
